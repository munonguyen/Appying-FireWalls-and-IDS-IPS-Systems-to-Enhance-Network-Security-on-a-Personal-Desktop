1. OBJECTIVE

The brute-force experiment evaluates how the Blue Force side can detect repeated authentication attempts and sudden connection spikes against exposed remote-access services. The defensive model combines Windows Defender Firewall with Snort IDS/IPS rules. The expected Blue Force behavior is:

- Detect repeated failed login attempts against SSH and RDP.
- Detect rapid request spikes from the same source IP.
- Classify the source as suspicious after a threshold is reached.
- Block the source using Windows Firewall or Snort inline IPS mode.
- Reduce or prevent successful brute-force access.

5. BLUE FORCE DETECTION LOGIC

The Blue Force detection approach is threshold-based. Instead of reacting to a single failed login, Snort rules look for repeated activity from the same source address within a short time window. This is suitable for brute-force detection because normal users may mistype a password once or twice, while brute-force tools produce repeated and predictable request patterns.

Configured or expected Snort rules:

- SID 3000002: SSH BRUTE-FORCE Possible Brute Force Attack Detected
  Threshold: 5 matching SSH events within 60 seconds
  Expected action: raise a high-priority SSH brute-force alert

- SID 3000003: SSH Failed Login Attempt
  Threshold: 3 matching password-attempt events within 30 seconds
  Expected action: raise a failed-login pattern alert

- SID 3000004: SSH RAPID Multiple Connections from Same Source
  Threshold: 10 new SSH connections within 60 seconds
  Expected action: raise a rapid-source SSH alert

- SID 3000006: RDP BRUTE-FORCE Multiple Connections Detected
  Threshold: 10 new RDP connections within 60 seconds
  Expected action: raise a high-priority RDP brute-force alert

- SID 3000007: RDP BRUTE-FORCE Rapid Connection Attempts
  Threshold: 20 new RDP connections within 30 seconds
  Expected action: raise a rapid RDP alert

6. BLOCKING RESPONSE DESIGN

The intended defensive workflow is:

1. Snort detects repeated login attempts or rapid connection bursts.
2. The attacker source IP is classified as suspicious after the threshold is reached.
3. The source IP is blocked using Windows Firewall or Snort inline IPS mode.
4. Later SSH/RDP attempts from that source are denied before authentication can continue.

Manual Windows Firewall blocking command after confirming the attacker IP:

New-NetFirewallRule `
  -DisplayName "Block brute-force source" `
  -Direction Inbound `
  -RemoteAddress <ATTACKER_IP> `
  -Action Block

Important reporting note:
In the current exported evidence, automatic blocking is not proven yet because firewall or IPS block logs are not included. The report should say that blocking is the response design, and only claim confirmed blocking after collecting Windows Firewall logs or Snort IPS drop logs.
